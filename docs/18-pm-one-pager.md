# WoofTennis — PM One-Pager (IA vNext)

## Цель
Снять UX-путаницу и запустить MVP с простой логикой: пользователь понимает, где смотреть текущие события и где создавать новые.

## Решение (зафиксировано)
- TabBar: только `Главная` и `Профиль`.
- `Главная`: единая временная сетка + primary CTA `Создать`.
- `Профиль`: настройки, кабинет тренера, архив уведомлений.
- `Уведомления` не в tabbar; оперативный канал остается Telegram.
- `Тренеры` и `Игра` временно убраны из нижней навигации как дублирующие слои.

## Базовая продуктовая логика
- Тренер создает событие (разовое/периодическое) на Главной.
- Назначение игрока:
  - `direct-attach` (игрок найден в базе),
  - `invite` (приглашение по ссылке).
- Игрок видит событие на Главной и получает TG-уведомление.
- Локации сквозные для обеих ролей; фильтр по локациям обязателен.

## Навигационная модель (source of truth для FE)
- Нижний tabbar всегда из 2 пунктов: `Главная`, `Профиль`.
- `Главная` — стартовый экран после входа; на экране всегда есть CTA `Создать`.
- `Создать` открывает единый flow `Создание события` (не отдельный tab).
- `Профиль` состоит из 3 секций: `Аккаунт и роль`, `Кабинет тренера`, `Уведомления`.
- После терминальных исходов (`invite_invalid`, `invite_expired`, `cancelled`) основной CTA ведет на `Главная`.

## Семантика статусов (MVP, source of truth для FE/QA)
- `draft` — событие создано, участник еще не назначен.
- `invited` — создан invite, ожидается действие игрока.
- `attached` — игрок назначен напрямую тренером.
- `accepted` — игрок подтвердил участие.
- `declined` — игрок отклонил приглашение.
- `cancelled` — событие отменено инициатором.
- `rescheduled` — событие перенесено на другой слот.
- `completed` — событие завершено.
- `invite_expired` — invite недействителен по сроку.
- `invite_invalid` — invite не существует или поврежден.
- `time_conflict` — действие отклонено из-за конфликта времени.

## Что меняется для команды

### Frontend
- Пересобрать shell навигации под 2 таба.
- Реализовать сценарий создания через CTA.
- Поддержать состояния `invite/direct-attach` и ошибки edge-cases.

### Backend
- Подтвердить event-centric endpoint-ы.
- Ввести единые коды ошибок для invite/attach/конфликтов времени.
- Гарантировать синхронность app-статуса и Telegram push.

### QA
- Прогнать 5 E2E сценариев (happy + negative).
- Проверить dual-role контекст и сохранение фильтров.
- Проверить соответствие статусов в UI и Telegram.
- Проверить 4 базовых состояния каждого критичного экрана: `loading`, `empty`, `error`, `success`.

### Design
- Обновить wireflow под event-centric терминологию.
- Зафиксировать состояния карточек и экранов (`loading`/`empty`/`error`/`success` + терминальные статусы invite/attach).
- Проверить читабельность RU-копирайта в узких экранах.

## Риски и контроль
- **Риск:** пользователи не находят вторичные функции без отдельных табов.  
  **Контроль:** явные входы в `Профиле` + CTA на Главной.
- **Риск:** invite/direct-attach дают неочевидные статусы.  
  **Контроль:** единый словарь статусов и терминальные экраны.
- **Риск:** расхождение Telegram и in-app статусов.  
  **Контроль:** атомарная запись статуса + отправка push в одном процессе.

## План внедрения
- Реализация идет одним контуром: навигация, Главная, CTA `Создать`, attach/invite, профиль-хаб, edge-cases и QA-матрица.
- FE и BE поставляют сразу полный e2e-флоу one-pager без phased-переключений в рантайме.

## UI copy expectations (MVP)
- Главная empty: `Пока нет событий` + CTA `Создать событие`.
- Успех `direct-attach`: `Игрок назначен` + CTA `К событию`.
- Успех `invite`: `Ссылка-приглашение готова` + CTA `Поделиться`.
- `invite_expired`: `Срок приглашения истек` + CTA `Запросить новый инвайт`.
- `invite_invalid`: `Приглашение недействительно` + CTA `На Главную`.
- `time_conflict`: `Это время уже занято` + CTA `Выбрать другое время`.

## Критерии готовности к старту разработки
- PM утвердил IA и терминологию сущностей.
- FE/BE согласовали контракт endpoint-ов и ошибок.
- QA получил E2E-матрицу и чеклист приемки.

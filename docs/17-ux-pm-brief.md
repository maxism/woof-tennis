# WoofTennis — UX/PM Brief (IA vNext)

## 1) Product Decision Snapshot

Принята упрощенная IA для MVP:
- TabBar: `Главная`, `Профиль`.
- `Главная` объединяет обзор и действие: временная сетка событий + primary CTA `Создать`.
- `Профиль` содержит настройки, кабинет тренера и архив уведомлений.
- Все оперативные уведомления дублируются в Telegram.

Удалены из таббара:
- `Тренеры` (временно);
- `Игра` (как отдельный раздел);
- `Уведомления` (перенесены в профиль).

## 2) Core UX Logic

### Главная
- Единая сетка времени: пользователь видит все свои актуальные события.
- Для dual-role: сегмент `Как игрок` / `Как тренер`.
- Фильтр по локациям обязателен.
- CTA `Создать` запускает сценарий создания события.

### Создание события
- Создатель (тренер/игрок) задает параметры события: время, локация, формат.
- Затем выбирает способ назначения участника:
  - `direct-attach`, если пользователь есть в базе;
  - `invite`, если нужен вход по ссылке.

### Профиль
- Настройки аккаунта и роли.
- `Кабинет тренера`: локации, расписание, отыгрыши.
- `Уведомления`: архив и история (для контроля, несмотря на TG-duplicate).

## 3) Словарь сущностей и статусов

### Сущности
- `Event`: базовая сущность события в календарной сетке.
- `Invite`: объект приглашения с кодом и сроком действия.
- `Attach`: результат прямого назначения пользователя в событие.
- `Location`: сквозная сущность, доступная игрокам и тренерам.

### Минимальные статусы UI
- `draft` — создано, не отправлено.
- `invited` — приглашение отправлено.
- `attached` — участник назначен напрямую.
- `accepted` — участие подтверждено.
- `declined` — приглашение отклонено.
- `cancelled` — событие отменено.
- `rescheduled` — событие перенесено.
- `completed` — событие завершено.

## 4) 5 E2E Scenarios (stress-tested)

1. **Coach_Create_DirectAttach**
   - Тренер создает событие -> выбирает игрока из базы -> игрок получает TG и видит событие на Главной.
   - Риск: конфликт времени у игрока.
   - UX-реакция: явный статус конфликта + предложение выбрать другое время.

2. **Coach_Create_Invite**
   - Тренер создает событие -> генерирует invite -> шэрит ссылку.
   - Риск: invite устарел/невалиден.
   - UX-реакция: понятный экран ошибки + CTA к запросу нового инвайта.

3. **Player_Accepts_Invite**
   - Игрок открывает ссылку -> видит детали -> подтверждает участие.
   - Риск: слот уже закрыт или отменен.
   - UX-реакция: терминальный статус + возврат на Главную.

4. **DualRole_ContextSwitch**
   - Пользователь переключает `Как игрок` / `Как тренер`.
   - Риск: потеря контекста фильтров и даты.
   - UX-реакция: сохранять фильтры и точку скролла в пределах сессии.

5. **Cancel_Reschedule_Notify**
   - Отмена или перенос события инициируется одной стороной.
   - Риск: разрыв между app-state и TG уведомлением.
   - UX-реакция: атомарный статус в app + синхронный TG push.

## 5) Decision Log (MVP)

- Низкая плотность таббара приоритетнее discoverability вторичных разделов.
- `Тренеры` как каталог исключен из MVP до валидации invite/direct-attach модели.
- `Игра` не выделяется в отдельную навигацию, чтобы не дублировать семантику Главной.
- Локации вынесены в сквозной объект для обеих ролей.

## 6) Role-based Backlog

### Frontend
- Обновить shell навигации и роутинг под IA vNext.
- Реализовать сценарии `direct-attach`/`invite` и статусные экраны.
- Добавить фильтр по локации в сетку.
- Перенести уведомления в профиль.

### Backend
- Поддержать event-centric API и проверку конфликтов времени.
- Добавить endpoint-ы attach/invite/accept c едиными error-кодами.
- Гарантировать синхронное создание in-app notification и TG push.

### QA
- Покрыть 5 E2E сценариев и негативные кейсы invite/attach.
- Проверить role-switch без потери контекста.
- Проверить консистентность статусов между app и Telegram.

### Design
- Обновить wireflow и состояния карточек под event-centric терминологию.
- Зафиксировать visual rules для статусов и пустых состояний.
- Провести readability-check на RU копирайте в узких ширинах.

## 7) Delivery Plan

### Wave 1: Core Flow
- Навигация, Главная, CTA `Создать`, базовый direct-attach/invite, профиль-хаб.
- Цель: рабочий end-to-end путь от создания до появления у игрока.

### Wave 2: Edge Cases and Quality
- Конфликты времени, невалидные инвайты, терминальные состояния.
- Полировка UX-статусов, QA-матрица, стабильность Telegram синхронизации.

## 8) Acceptance Gates

- PM утверждает IA и терминологию сущностей.
- FE/BE подтверждают совместимый контракт endpoint-ов и ошибок.
- QA подтверждает прохождение E2E smoke для 5 ключевых сценариев.

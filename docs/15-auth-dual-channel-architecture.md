# Ревью архитектуры авторизации: веб (Login Widget) и Mini App

**Статус:** архитектурное решение и задания для разработки.  
**Контекст:** продуктовая цель — **два канала входа** с одним и тем же пользователем в БД и одним типом сессии (JWT).

**Синхронизация с ревью бэкенда:** текущий код = ветка Mini App (первый шаг). До целевой модели в **коде** (§4.2, §5–6) не хватает второго эндпоинта, второго криптовалидатора, rate limit, тестов. **Документация** по виджету и `/setdomain` уже в `docs/03-api-spec.md`, `docs/14-onboarding-setup.md`, целевая структура Auth — в `docs/06-backend-structure.md` (см. §3.1, §4.1).

---

## 1. Целевая логика (как должно быть)


| Канал        | Где открывается                 | Механизм Telegram                                                                                         | Ожидаемый UX                                                                                                                                       |
| ------------ | ------------------------------- | --------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Веб-сайт** | Обычный браузер                 | [Login Widget](https://core.telegram.org/widgets/login)                                                   | Кнопка «Войти через Telegram» → виджет / редирект → данные пользователя + `hash` → сервер проверяет подпись → выдаётся JWT                         |
| **Mini App** | WebView внутри клиента Telegram | [Web Apps / initData](https://core.telegram.org/bots/webapps#validating-data-received-via-the-mini-app) | Открытие приложения → `window.Telegram.WebApp.initData` уже есть → фронт шлёт строку `initData` на API → сервер валидирует → JWT без лишних кликов |


Обе ветки должны приводить к **одному** идентификатору пользователя (`telegramId` / запись в `users`) и **одному** формату ответа сессии (`accessToken` + `user`), чтобы остальной API не ветвился по способу входа.

---

## 2. Важно: это два разных криптографических протокола

Путаница обычно возникает здесь: оба варианта используют токен бота и HMAC, но **секрет и состав полей различаются**.

- **Mini App `initData`:** секретный ключ строится как `HMAC-SHA256(key="WebAppData", message=bot_token)` (байты), затем HMAC по `data_check_string` из полей `initData`. Описано в документации Web Apps.
- **Login Widget:** проверка авторизации по [официальному алгоритму виджета](https://core.telegram.org/widgets/login#checking-authorization): `secret_key = SHA256(bot_token)`, затем `hash = HMAC-SHA256(secret_key, data_check_string)` по **набору полей виджета** (не та же строка, что у Mini App).

**Вывод для реализации:** на бэкенде нужны **две отдельные функции валидации** (или один модуль с двумя явными входами), а не попытка скормить ответ виджета в валидатор `initData`.

## 5. Рекомендуемая целевая архитектура

### 5.1 Принципы

1. **Один пользователь** — ключ `telegramId` (как сейчас в модели данных).
2. **Два публичных способа установить сессию** — оба заканчиваются выдачей того же JWT и того же DTO `user`.
3. **Разделение эндпоинтов** (рекомендация для ясности и типобезопасности DTO):
  - `POST /auth/telegram/mini-app` — тело `{ "initData": string }` *(можно оставить текущий `/auth/telegram` как алиас на переходный период)*;
  - `POST /auth/telegram/widget` — тело с полями, которые возвращает виджет (`id`, `first_name`, `auth_date`, `hash`, …), либо одна строка query, если выберете редирект-режим — см. оф. документацию виджета.
4. **BotFather:**
  - Mini App URL — как сейчас.
  - Для виджета — привязка **домена сайта** к боту ([Login Widget — настройка](https://core.telegram.org/widgets/login)).

### 5.2 Фронтенд: определение контекста

- Если доступен `window.Telegram?.WebApp?.initData` (непустой) — канал **Mini App**, бесшовный вход как сейчас.
- Иначе — канал **веб**, показываем **Login Widget**, по колбэку отправляем данные на `POST /auth/telegram/widget`.

### 5.3 Хранение JWT

- Mini App: по желанию CloudStorage + память (как в `07`).
- Веб: `localStorage` / sessionStorage + те же заголовки `Authorization` — единый HTTP-клиент уже может подставлять Bearer.

## 6. Задания бэкенд-разработчику

1. **Спецификация:** контракт `POST /auth/telegram/widget` уже в `docs/03-api-spec.md` — при изменении API править `03` синхронно с кодом.
2. **Реализация:** функция проверки подписи по [алгоритму Login Widget](https://core.telegram.org/widgets/login#checking-authorization), **не** смешивать с `initData`.
3. **Общая логика:** после успешной проверки — тот же `upsert` пользователя по `telegramId` и та же генерация JWT, что и для Mini App (вынести общий метод в `AuthService`).
4. **Безопасность:** лимиты на частоту запросов к `/auth/*`, логирование неуспешных проверок без утечки токена бота.
5. **Конфигурация:** для продакшена — домен виджета зарегистрирован через `/setdomain` (шаги в `docs/14-onboarding-setup.md`); при смене домена обновлять BotFather и проверки.
6. **Тесты:** юнит-тесты на обе функции валидации с тестовыми векторами из документации Telegram (или сфабрикированными известными парами).

---

## 7. Задания фронтенд-разработчику

1. **Детект окружения:** явный хелпер `isTelegramMiniApp(): boolean` (по наличию и непустоте `initData` / флагу платформы SDK).
2. **Mini App:** оставить текущий поток `initData` → существующий эндпоинт (при переименовании — обновить путь).
3. **Веб:** встроить Login Widget (скрипт Telegram), обработать `onAuth` / редирект — отправить полученные поля на новый эндпоинт, сохранить JWT так же, как после Mini App (единый `authStore`).
4. **UI:** экран входа в браузере — только виджет (и пояснение пользователю); не предлагать «вставьте initData вручную».
5. **Документация фронта:** обновить `docs/05-frontend-structure.md` — две ветки авторизации в диаграмме и в описании `authStore` / `useBootstrapAuth`.

## 8. Что не нужно путать с авторизацией

- **Webhook бота** — про входящие апдейты Telegram (команды, кнопки), а не про логин пользователя на сайте.
- **Deep link `t.me/bot?startapp=...`** — про открытие Mini App, а не про Login Widget на произвольном домене.
